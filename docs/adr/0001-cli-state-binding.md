# ADR 0001: CLI State Binding Architecture for Network Visualizer

* **สถานะ**: เสนอแล้ว (Proposed)
* **ผู้มีส่วนร่วมตัดสินใจ**: Antigravity AI, ผู้ใช้งาน
* **วันที่ตัดสินใจ**: 2026-07-10

---

## 1. บริบทและปัญหา (Context and Problem Statement)

ในระบบเรียนรู้ Cisco Learn Portal เราต้องการแสดงแผนภาพความเชื่อมโยงเครือข่ายทางกายภาพ (Network Diagram) ควบคู่กับระบบจำลองคำสั่ง CLI (CLI Terminal Simulator) เมื่อผู้เรียนพิมพ์คำสั่ง เช่น `shutdown` หรือตั้งค่าเปลี่ยน VLAN พอร์ตหรือสายในแผนภาพเครือข่ายจะต้องตอบสนองเปลี่ยนสีตามทันที

หากเราใช้วิธีเปลี่ยนสไตล์ DOM ตรง ๆ ภายในฟังก์ชันการถอดความหมายคำสั่ง (`executeCommand()` ใน `visualizer.js`) จะก่อให้เกิดการผูกมัดรหัสที่แน่นหนาเกินไป (High Coupling) และยากแก่การดูแลรักษาหรือสลับไดอะแกรมอื่นในบทเรียนอื่น ๆ

---

## 2. ทางเลือกที่พิจารณา (Considered Options)

### ทางเลือกที่ A: Direct DOM Manipulation (เขียนคำสั่งคุมภาพลงในตัวแปรแยกตรง ๆ)
* **การทำงาน**: เมื่อ Parser CLI ตีความคำสั่งเสร็จ ให้เรียก `document.getElementById('fa01-link').style.stroke = 'red'` ทันที
* **ข้อดี**: พัฒนาง่าย ได้ผลลัพธ์เร็ว ไม่ต้องการสถาปัตยกรรมซับซ้อน
* **ข้อเสีย**: ฟังก์ชันตีความคำสั่ง (CLI Parser) จะปนเปื้อนด้วยโค้ดจัดการภาพ แสดงผลในลักษณะสปาเก็ตตี้ หากมีการเปลี่ยนแผนภาพใหม่ในบทถัดไปจะต้องรื้อฟังก์ชัน `executeCommand()` ใหม่หมด

### ทางเลือกที่ B: State Observer Event Broker (การใช้ระบบแจ้งเตือนการเปลี่ยนสถานะแบบกระจาย)
* **การทำงาน**: CLI Simulator จะทำหน้าที่ปรับเปลี่ยนสถานะภายใน (State Object) เท่านั้น และจะยิง Event ย่อยออกมา (เช่น `"cli-state-changed"`) ผ่านการระบุประเภทคุณลักษณะที่เปลี่ยนแปลง จากนั้นตัววาดแผนภาพ (Renderer) จะคอยฟัง Event นี้แล้วดึงข้อมูลสถานะล่าสุดไปอัปเดตวาดภาพใหม่
* **ข้อดี**: 
  * แยก Logic และ UI ออกจากกันโดยเด็ดขาด (Decoupled)
  * CLI Simulator สามารถนำกลับมาใช้ใหม่ได้โดยไม่ต้องผูกกับรูปภาพเดิม
  * สามารถสลับไดอะแกรมเครือข่ายของแต่ละบทเรียนได้โดยไม่ต้องเขียน CLI Parser ใหม่
* **ข้อเสีย**: ต้องใช้โค้ดเกริ่นนำระบบส่งสาร (Event Broker) เพิ่มขึ้นเล็กน้อย

---

## 3. ผลการตัดสินใจ (Decision Outcome)

เลือก **ทางเลือกที่ B: State Observer Event Broker** เนื่องจากเป็นจุดเปลี่ยนที่ส่งผลต่อการเขียนระบบแสดงภาพ Network Diagram ในทุก ๆ บทเรียนถัดไป การแยกโมดูลนี้ช่วยให้สามารถพัฒนาไดอะแกรมเครือข่ายเฉพาะเจาะจงของแต่ละบทเรียน (เช่น ไดอะแกรม VLAN ในบทที่ 8 และไดอะแกรม Password Recovery ในบทที่ 9) โดยแชร์ตัวจำลอง CLI ตัวเดียวกันได้สำเร็จ

### รายละเอียดข้อกำหนดการแสดงผล (UI Specifications accepted)
1. **การตั้งค่า VLAN (VLAN Membership)**: ใช้รูปแบบ **ผสมผสาน** คือ สายเคเบิลของคอมพิวเตอร์ที่ต่อกับพอร์ตนั้น ๆ จะเปลี่ยนสีตามรหัสสีของ VLAN และแสดง **VLAN ID Badge** (เช่น `[V10]`) ปรากฏกำกับข้างสายเชื่อมต่อ
2. **สถานะพอร์ตการเชื่อมต่อ (Port Status)**: ใช้รูปแบบ **ผสมผสาน** คือ เมื่อสั่ง `shutdown` หรือ `no shutdown` สายเคเบิลเชื่อมต่อจะเปลี่ยนสี (แดง = Down, เขียว/VLAN = Up) พร้อมกับมีไฟสถานะ **Port LED Indicator** ขนาดเล็กบนตัวสวิตช์จำลองสลับสีแดง/เขียวเพื่อความสมจริงเชิงฮาร์ดแวร์

### ผลกระทบและผลลัพธ์ (Consequences)
* **ข้อดี**:
  * ตัวแปลภาษา CLI (`visualizer.js` L646+) จะไม่มีโค้ดจัดการ UI ปนอยู่เลย ทำให้ง่ายต่อการตรวจทานความถูกต้องด้านวิชาการ
  * การขยายโครงสร้างไดอะแกรมในอนาคตทำได้ทันทีโดยเพิ่มฟังก์ชัน Render
* **ข้อเสีย**:
  * โครงสร้างของ `state` จำเป็นต้องอัปเดตค่าให้ครบถ้วนทุกครั้งก่อนกระจาย Event เพื่อป้องกันไม่ให้ UI ฝั่งรับแสดงผลสถานะตกหล่น
